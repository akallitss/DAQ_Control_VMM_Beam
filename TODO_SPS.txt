 1. base_data_dir                                    run_config_beam.py (SITES)
--------------------------------------------------------------------------------
 What : directory on the DAQ machine's DATA disk where runs/ analysis/ etc.
        are created. Keep the trailing slash.
 How  : find the big data partition:
            df -h                      # look for the large, mostly-empty disk
            lsblk -f                   # partition layout
        Create the tree and make sure your user can write:
            mkdir -p /local/p2/p2data/sps_vmm_2026 && touch /local/p2/p2data/sps_vmm_2026/.w
        Also copy config/detectors/ contents there (define_detectors etc. are
        read from <base_data_dir>/config/detectors/).

 2. daq_host                                         run_config_beam.py (SITES)
--------------------------------------------------------------------------------
 What : IP where daq_control/flask reach the hv/lv/daq servers.
 How  : everything runs on the SAME machine -> leave 127.0.0.1. Done.
        (Only change it if services are ever split across machines; then use
        `hostname -I` on the server machine.)
        Separately: to open the GUI from the control room you need the DAQ
        machine's LAN IP — `hostname -I` or `ip -br addr` — and port 5002
        reachable (check any local firewall: `sudo ufw status` /
        `sudo firewall-cmd --list-ports`).

 3. hv_ip  (CAEN mainframe)                          run_config_beam.py (SITES)
--------------------------------------------------------------------------------
 What : IP of the CAEN HV crate on the network the DAQ machine sees.
 How  , in order of reliability:
        a) Read it on the crate itself: front-panel display/menu shows the
           LAN settings (or plug in the local keypad/screen).
        b) If it was configured before, it is often written on a label/tape
           on the crate, or known by whoever set up the crate (Dream DAQ
           uses the same crate family — check DAQ_Control_Dream_Beam's
           run_config_beam.py for the value already in use at SPS).
        c) Scan the crate subnet from the DAQ machine:
               sudo arp-scan --interface=<nic> 192.168.10.0/24
               # or:  nmap -sn 192.168.10.0/24
           CAEN mainframes show a CAEN OUI in the MAC vendor column.
        Test:  ping <hv_ip>   and log in once with the CAEN GECO/web tool.

 5. lv_units  (Aim-TTi PSU IPs)                      run_config_beam.py (SITES)
--------------------------------------------------------------------------------
 What : LAN IPs of the two TTi bench supplies (tti1, tti2).
 How  : a) On each unit's front panel: Menu/Utilities -> Interface -> LAN
           shows the current IP (they may be DHCP — consider setting a
           static IP or a DHCP reservation so it survives power cycles).
        b) LXI discovery from the DAQ machine:
               lxi discover            # apt install lxi-tools
        c) Subnet scan as in item 3 (vendor "Aim-TTi"/"TTi").
        Test SCPI directly (port 9221 is fixed on these units):
            nc -zv <ip> 9221
            printf '*IDN?\n' | nc -q1 <ip> 9221      # should return the IDN string
        Note which physical rail is on which unit+channel and set
        lv_info['units'][name]['channels'] if not all 3 channels are used.

 6. interfaces  (capture NICs + ALINX config)        run_config_beam.py (SITES)
--------------------------------------------------------------------------------
 What : Linux name of each NIC that receives VMM UDP, and the absolute path
        of the alinx-sc JSON config on the DAQ machine.
 How  : list NICs and see which has a link up on the readout fibre/cable:
            ip -br link                 # UP/DOWN per interface
            ethtool <iface>             # link detected: yes
        Confirm data actually arrives on it (with the front-end sending):
            sudo tcpdump -i <iface> -nn udp -c 5
        The UDP *source* IP printed there is the FEC/ALINX — the QA
        auto-detects it, nothing to configure, but it is a good check.
        ALINX config: locate the file used to arm the firmware:
            locate config_alinx_noThresholds.json   # or: find / -name 'config_alinx*' 2>/dev/null
        and test the slow control end to end:
            alinx-sc --config-file <path> --read-link-status
        SRS chain: if also capturing SRS, uncomment the 'srs' entry and set
        its iface the same way (slow_control: 'none').

 10. Backup to EOS                                   backup_config.py
--------------------------------------------------------------------------------
 What : EOS_DIR, CERN_PRINCIPAL, GPG_PASS_FILE.
 How  : EOS_DIR — the experiment's EOS space, e.g. an existing
            /eos/project/.../p2_sps_beam/ or your user space
            /eos/user/<initial>/<username>/p2_sps_beam/  (create it via
            lxplus or CERNBox first). It must be FUSE-mounted on the DAQ
            machine (ls /eos/... works).
        CERN_PRINCIPAL — your CERN account, e.g. akallits@CERN.CH
            (check with:  klist  after a kinit on lxplus).
        GPG_PASS_FILE — store your CERN password encrypted:
            echo -n '<password>' | gpg --encrypt -r <your-gpg-key> -o ~/.cern_pass.gpg
        Test the chain once:
            gpg -d ~/.cern_pass.gpg | kinit <principal>
            rsync -a --dry-run <base_data_dir>/config/ /eos/<...>/config/
